# Briefings & Knowledge Design Spec

> **Note:** Originally designed as "Memory Tiers". Renamed to "Briefings & Knowledge" on 2026-03-27.
> Implementation: `codespecs/briefings.md`. Briefings live in `briefings/`, knowledge at project root `knowledge.md`.

## Problem

The Chief of Staff agent lacks context from meetings. When planning pipelines or answering questions, it doesn't know about recent decisions, priority shifts, or key numbers discussed in meetings — forcing the user to re-explain. There's no structured path for meeting knowledge to become permanent institutional knowledge.

## Solution

A briefings + knowledge system fed by meeting transcripts from Google Drive. Briefings capture recent meeting extracts (ephemeral, 4-week retention). Knowledge stores validated permanent facts (requires user approval). Topic-based compaction is deferred.

## Source

Meeting transcripts generated by Otter/Fireflies, stored in Google Drive:
`https://drive.google.com/drive/folders/1A9G0sBOJdAK-qafaDymuZhg3frJnakLs`

## Tier 1: Short-Term Memory

**Purpose:** Raw structured extracts from individual meetings.
**Location:** `briefings/recent/`
**Retention:** 2 weeks, then archived or deleted during weekly compaction.
**Update cadence:** Every meeting, triggered by user ("process this meeting" + link/filename).

### Ingestion Process

1. User provides transcript link or filename from the Drive folder
2. CoS reads transcript via `gws drive` commands
3. CoS extracts structured summary into `briefings/recent/YYYY-MM-DD-{meeting-slug}.md`
4. Updates `briefings/briefings.md` index

### Extraction Categories

- **Decisions** — what was decided and by whom
- **Action items** — owner, deadline, description
- **Key numbers/metrics** — with context (what they measure, direction, comparison)
- **Strategy/priority shifts** — what changed and why

### File Format

```markdown
---
date: 2026-03-24
meeting: Pipeline Review
attendees: [Nish, Sarah, Mike]
expires: 2026-04-07
---
## Decisions
- Pushing healthcare deals to Q3; not ready for Q2 close

## Action Items
- [ ] Sarah: Update forecast model by Friday 2026-03-27
- [ ] Mike: Pull churn data for QBR prep

## Key Numbers
- Q2 pipeline at $2.1M (down from $2.4M last week)
- 3 deals slipped from March to April

## Strategy/Priority Shifts
- Focus shifting from new logo to expansion for rest of Q2
```

## Tier 2: Medium-Term Memory

**Purpose:** Weekly topic-centric synthesis across multiple meetings. The key transformation: meeting-centric → topic-centric.
**Location:** `briefings/medium-term/` *(deferred)*
**Retention:** ~2 months (8 weekly summaries). Older summaries archived.
**Update cadence:** Weekly, triggered by user or CoS prompt ("You have 5 uncompacted meetings. Want me to run weekly compaction?").

### Compaction Process

1. CoS reads all short-term files from the past week
2. Groups extracted content by topic/theme (not by meeting)
3. Writes `briefings/medium-term/YYYY-WXX-summary.md`
4. Expires short-term files older than 2 weeks (archive or delete)
5. Identifies knowledgebase candidates (see Tier 3)

### File Format

```markdown
---
week: 2026-W13
meetings_compacted: 4
sources: [2026-03-23-leadership-sync, 2026-03-24-pipeline-review, ...]
---
## Pipeline & Revenue
- Q2 pipeline dropped to $2.1M; 3 deals slipped March→April
- Decision: shift focus from new logo to expansion for rest of Q2
- Sarah updating forecast model (due 2026-03-27)

## Customer Success
- Churn risk flagged for 2 mid-market accounts
- QBR prep underway — Mike pulling churn data

## Hiring
- No new headcount approved; revisit in Q3 planning

## Open Action Items
- [ ] Sarah: forecast model update (due 2026-03-27)
- [ ] Mike: churn data for QBR (due 2026-03-28)
```

## Tier 3: Knowledgebase (Long-Term)

**Purpose:** Validated, permanent institutional knowledge. The source of truth.
**Location:** `knowledge.md` (existing file, extended).
**Retention:** Permanent until explicitly updated or removed.
**Update cadence:** Weekly during compaction. **Requires explicit user approval for every addition.**

### What Belongs Here

- Business rules ("we don't discount below 20%")
- Org structure / ownership ("Sarah owns the forecast model")
- Strategic decisions ("entering healthcare vertical in Q3")
- Process definitions ("QBRs happen first week of quarter")
- Data model facts ("ARR includes expansion but not services")

### Promotion Process

1. During weekly compaction, CoS identifies candidates — items that appear stable, permanent, and referenceable
2. CoS presents candidates to user: "I'd like to add these to the knowledgebase: [list]. Approve/reject each?"
3. Approved items added to `knowledge.md` under appropriate sections
4. User can also manually promote anytime: "add X to the knowledgebase"

### What Does NOT Belong

- Ephemeral numbers (this week's pipeline total)
- In-progress action items
- Anything that changes week to week

## Memory Index: MEMORY.md

`briefings/briefings.md` serves as the lightweight index for all tiers. Updated on every ingestion and compaction.


## How Memory Gets Used

### Cold Start (every conversation)

Current cold start reads `briefings/briefings.md`, `briefings/active-work.md`, `knowledge.md`.

New behavior:
1. Read `briefings/briefings.md` index — see what's available across all tiers
2. Read `knowledge.md` — permanent KB (unchanged)
3. Read latest 2 medium-term summaries — recent topic context
4. Scan short-term file names/dates — only read full content if topic matches current request

### Proactive Surfacing During Planning

When CoS plans a pipeline or analysis:
1. Match request keywords against medium-term topic headings and short-term meeting slugs
2. Read matching files for relevant detail
3. Inline context into the plan: "Note: In the 2026-03-24 pipeline review, Q2 pipeline was reported at $2.1M with 3 slipped deals."
4. Include in subagent dispatch prompts via Context Inlining

### On-Demand Recall

When user asks "what did we decide about X?":
1. Search KB first (most authoritative), then medium-term, then short-term
2. Return answer with source: "Per the 2026-03-24 pipeline review: decision was to shift to expansion focus for Q2."

### Subagent Access

Subagents receive memory context through dispatch prompts — they don't read memory files directly. CoS decides what's relevant and inlines it. This keeps subagents focused and avoids them reading the entire memory tree.

## Directory Structure

```
knowledge.md                  # Permanent validated KB (project root)
briefings/
├── briefings.md              # Index of recent briefings
├── active-work.md            # Task state (separate concern, unchanged)
├── recent/                   # Meeting extracts (4-week retention)
│   ├── 2026-03-24-pipeline-review.md
│   └── 2026-03-23-leadership-sync.md
└── archive/                  # Expired briefings (optional)
```

**Git tracking:**
- `knowledge.md` — tracked (permanent, valuable)
- `short-term/`, `medium-term/` — tracked (want history and cross-session access)
- `active-work.md` — gitignored (session-ephemeral, existing behavior)
- `archive/` — gitignored (expired, only kept for reference)

## Verification

1. **Ingestion test:** Process a real transcript → verify short-term file has correct structure and content
2. **Compaction test:** With 3+ short-term files, run compaction → verify medium-term summary is topic-centric, not meeting-centric
3. **KB promotion test:** During compaction, verify candidates are surfaced and only approved items land in knowledge.md
4. **Expiry test:** After 2 weeks, verify short-term files are archived
5. **Proactive surfacing test:** Start a sales pipeline → verify relevant meeting context appears in plan
6. **Recall test:** Ask "what did we decide about X?" → verify correct answer with source attribution
